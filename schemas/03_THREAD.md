# Thread Schema

Version: 1.0
Status: Locked
Last Updated: 2026-07-07

---

# Purpose

The Thread Schema defines the primary content object created by Aura AI.

It represents a complete Threads publication generated through the Write feature and serves as the foundation for drafting, scheduling, publishing, planning, analytics, and AI analysis.

Every Thread exists as an independent business object and contains all information required to preserve its content, generation context, and publishing lifecycle.

---

# Definition

A Thread is a complete collection of individual posts that together form a single Threads publication.

Unlike an individual post, a Thread represents the entire content structure generated by Aura AI.

Depending on the selected writing format, a Thread may contain:

- Four posts (H → E → R → O)
- Five posts (H → E → R1 → R2 → O)

Regardless of the number of individual posts, Aura AI treats the complete collection as one Thread.

---

# Scope

The Thread Schema is created exclusively by the Write feature.

Once created, it becomes available throughout Aura AI and may be used by:

- Write
- Planner
- Stats
- Draft
- Analysis

The Thread Schema does not include engagement replies generated by Scout.

Replies are represented separately by the Reply Schema.

---

# Core Concept

Aura AI distinguishes between an individual post and a complete Thread.

```
Thread
│
├── Post 1 (Hook)
├── Post 2 (Educate)
├── Post 3 (Relate)
└── Post 4 (Outcome)
```

or

```
Thread
│
├── Post 1 (Hook)
├── Post 2 (Educate)
├── Post 3 (Relate Part 1)
├── Post 4 (Relate Part 2)
└── Post 5 (Outcome)
```

The complete structure above represents a single Thread object.

Every generated Thread also preserves the generation context used during creation, including the selected writing mode, language, tone, topic, word length, product link, and other creator-defined settings.

This allows Aura AI to preserve the original intent and configuration of every generated Thread throughout its lifecycle.

# Schema Overview

The Thread Schema represents a complete Threads publication generated by Aura AI.

Each Thread is self-contained, preserving both its generated content and the generation context used to create it.

Rather than storing only the final text, the Thread Schema captures everything required to understand, display, manage, and publish the Thread throughout its lifecycle.

---

# Core Properties

Each Thread contains the following core information:

| Property | Description |
|----------|-------------|
| Thread ID | Unique identifier for the Thread. |
| Account Context ID | The Account Context that owns the Thread. |
| Topic | Creator-defined topic used for generation. |
| Writing Mode | Selected writing mode. |
| Language | Language used during generation. |
| Tone | Selected writing tone. |
| Word Length | Selected thread length preference. |
| Product Link | Product or landing page associated with the Thread (if provided). |
| Link in Bio | Indicates whether the generated Thread should reference the creator's bio link. |
| Thread Structure | Defines whether the Thread follows H→E→R→O or H→E→R1→R2→O. |
| Posts | Ordered collection of individual posts belonging to the Thread. |
| Current Status | Current lifecycle state of the Thread. |
| Generated At | Date and time the Thread was generated. |
| Last Modified At | Date and time the Thread was last edited by the creator. |

These properties describe both the generated content and the creator's generation choices.

# Relationships

The Thread Schema is owned by the active Account Context and serves as the central business object for multiple Aura AI features.

Relationship overview:

```
Account Context
        │
        ▼
     Thread
        │
 ┌──────┼───────────┬───────────┐
 ▼      ▼           ▼           ▼
Draft Planner     Stats     Analysis
```

The Write feature creates the Thread.

After creation, other business features interact with the same Thread object instead of creating separate copies.

Examples include:

- Draft stores the Thread for future editing.
- Planner organises the Thread for future publication.
- Stats records performance after publication.
- Analysis stores AI evaluation related to the Thread.

This relationship model ensures every stage of the Thread lifecycle references the same business object while preserving a single source of truth.

---

# Lifecycle

Every Thread follows a clear lifecycle from generation to publication.

Unlike temporary AI responses, a Thread is a persistent business object that may continue evolving until it is published.

The typical lifecycle is:

```
Generate
     │
     ├──────────────┐
     ▼              ▼
Publish Now     Save Draft
                     │
                     ▼
                 Edit Draft
                     │
                     ├──────────────┐
                     ▼              ▼
               Publish Now     Schedule
                                    │
                                    ▼
                               Published
```

Not every Thread follows the same path.

A creator may:

- Publish immediately after generation.
- Save as a Draft for future editing.
- Schedule the Thread for automatic publication.

Regardless of the chosen workflow, every published Thread reaches the same final state.

---

# Ownership

Every Thread belongs to exactly one Account Context.

Ownership hierarchy:

```
Aura User
      │
      ▼
Account Context
      │
      ▼
   Thread
```

The Thread never belongs directly to the Aura User.

Instead, it is permanently associated with the Account Context that generated it.

This ownership never changes throughout the Thread's lifecycle.

---

# Lifecycle Behaviour

A Thread remains editable until it is published.

Creators may:

- Modify any individual post.
- Save updated Drafts.
- Change scheduled publication dates.
- Publish immediately.

Once a Thread has been published:

- Its content becomes read-only.
- It is stored as published history inside Planner.
- It becomes available for performance tracking in Stats.
- Future edits require creating a new Thread rather than modifying the published one.

Publishing marks the completion of the Thread lifecycle.

The published Thread remains permanently available as historical business data for reporting, planning history, and performance analysis.

# Business Rules

The Thread Schema follows a set of rules that ensure every generated Thread remains consistent throughout its lifecycle.

## Rule 1 — One Thread Represents One Complete Publication

Each Thread represents one complete Threads publication.

A Thread consists of multiple individual posts organised into a single logical content unit.

Supported structures include:

- H → E → R → O
- H → E → R1 → R2 → O

Regardless of the number of individual posts, the complete collection is always treated as one Thread.

---

## Rule 2 — Thread Preserves Generation Context

Every Thread permanently stores the generation context used during creation.

This includes:

- Topic
- Writing Mode
- Language
- Tone
- Word Length
- Product Link
- Link in Bio preference

Changes made to the User Profile after generation must never modify the original Thread.

---

## Rule 3 — One Owner Per Thread

Each Thread belongs to exactly one Account Context.

A Thread cannot be transferred between different Account Contexts.

All Drafts,

Planner activities,

Statistics,

and Analysis records associated with the Thread continue referencing the same Account Context throughout its lifecycle.

---

## Rule 4 — Published Threads Are Immutable

Once a Thread has been published:

- The content becomes read-only.
- The Thread is recorded as publishing history.
- Performance data begins accumulating.
- The Thread cannot return to Draft or Scheduled status.

If the creator wishes to publish an updated version, a new Thread must be generated or created.

---

## Rule 5 — Thread Is the Single Source of Truth

The Thread remains the authoritative source for all generated content.

Business features such as:

- Draft
- Planner
- Stats
- Analysis

must always reference the same Thread rather than creating duplicate copies.

This ensures every feature operates on a consistent representation of the creator's content.

---

# Validation Rules

Before a Thread is saved, scheduled, or published, Aura AI should validate that:

- A valid Account Context is active.
- The Thread contains at least one post.
- The Thread follows a supported structure.
- Required generation properties are present.
- The Thread is eligible for the requested operation.

Before publishing, Aura AI should additionally verify that:

- The Thread has not already been published.
- The target publishing time is valid (for scheduled content).
- The connected Threads account is available for publishing.

If validation fails, the requested operation should not proceed until the issue has been resolved.

These validation rules ensure every Thread remains complete, publishable, and consistent throughout its lifecycle.

# Business Examples

The following examples demonstrate how the Thread Schema behaves during normal creator workflows.

### Example 1 — Generate and Publish Immediately

A creator opens the Write feature and enters:

- Topic: Pinterest Affiliate Marketing
- Writing Mode: Educational
- Language: English
- Tone: Professional
- Structure: H → E → R → O

Aura AI generates one complete Thread containing four individual posts.

The creator reviews the content and selects **Publish Now**.

The Thread immediately enters the Published state and becomes available in:

- Planner (Publishing History)
- Stats (Performance Tracking)
- Analysis (Content Analysis)

---

### Example 2 — Save as Draft

A creator generates a Thread but decides it is not ready for publication.

Instead of publishing immediately, the creator selects **Save Draft**.

The Thread remains editable.

The creator may:

- Edit any post.
- Update the product link.
- Modify the content.
- Publish later.
- Schedule for future publication.

Throughout this process, the same Thread object is preserved.

---

### Example 3 — Schedule for Future Publication

A creator generates a Thread and schedules it for publication next Monday.

The Thread remains unchanged until the scheduled publishing time.

Once successfully published:

- The Thread becomes read-only.
- Planner moves it from Scheduled to Publishing History.
- Stats begin collecting performance metrics.
- Analysis remains associated with the published Thread.

No duplicate Thread is created during this transition.

# Design Rationale

The Thread Schema serves as the central business object within Aura AI.

Instead of creating separate copies of content for Drafts, Planner, Statistics, or Analysis, every feature references the same Thread throughout its lifecycle.

This approach provides:

- A single source of truth.
- Consistent business ownership.
- Reduced data duplication.
- Simplified feature integration.
- Better long-term maintainability.

The Thread also preserves the creator's original generation context, ensuring that every published Thread can always be understood exactly as it was originally created.

# Final Statement

The Thread Schema represents the core content object of Aura AI.

Every Thread captures both the creator's intent and the AI-generated outcome, preserving them as a complete, self-contained business object throughout its lifecycle.

From generation to drafting, scheduling, publication, historical tracking, and performance analysis, the Thread remains the single source of truth for creator content.

As Aura AI grows, the Thread Schema continues to provide a stable and consistent foundation upon which every content-related feature is built.
